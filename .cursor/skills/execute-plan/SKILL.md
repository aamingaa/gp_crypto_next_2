# Minimal Project Refactor

## 用途

当需要修改当前仓库，并且目标是以下几类时，使用这个技能：

- 尽量减少不必要改动
- 遵循当前项目结构和已有编码风格
- 优先在现有逻辑上扩展，而不是额外引入一套新框架
- 按任务清单执行，并在完成后打勾

## 工作原则

1. 优先做小范围、本地化修改，而不是大面积重写。
2. 尽可能复用已有模块、辅助函数、命名方式和数据流。
3. 贴近当前被修改文件已经在使用的风格：
   - 命名风格
   - 注释密度
   - 函数结构
   - logging / print 习惯
4. 除非当前代码确实需要，否则不要额外抽象出新的层级。
5. 如果有多种实现方案，优先选择对当前项目影响最小的方案。

## 修改前检查

开始改代码前，先做这些事：

- 找出本次需求必须改动的最小文件集合
- 先读附近代码，模仿局部风格，而不是自创一套风格
- 检查是否已有类似逻辑，优先在现有逻辑上扩展
- 避免顺手改动无关行为

## 执行清单

执行时复制并更新下面这份清单：

```markdown
任务进度：
- [ ] 明确本次需求的精确范围
- [ ] 确认最小改动文件集合
- [ ] 检查周边代码风格和现有实现模式
- [ ] 在进行较大改动前先更新计划/checklist
- [ ] 实现最小可行修改
- [ ] 验证修改后的流程仍符合当前项目行为
- [ ] 检查涉及文件的 lint / 运行问题
- [ ] 勾选已完成项，并总结风险与后续事项
```

## 修改规则

- 优先修改已有函数，而不是平行再造一套新函数。
- 优先做兼容现有配置的改动，而不是直接硬编码改变行为。
- 除非理由充分，否则不要调整当前文件结构。
- 注释保持简洁，并与仓库当前风格一致。
- 除非用户明确要求，否则不要做大规模重命名、全局格式化或顺手清理。
- 如果项目里已有计划文件，执行过程中要同步更新对应 checklist。

## 输出要求

汇报进度或完成情况时，应包含：

1. 改了什么
2. 有哪些内容是刻意不改的
3. 兼容性或验证情况
4. checklist 的勾选进度

## 适用示例

- “在当前项目里小改一下这段逻辑，别重构太多”
- “按现有风格接入一个新指标”
- “尽量少动代码，修复这个 bug”
- “先按计划逐项执行并打勾”

## 不适用场景

- 从零开始的大重构
- 未经用户确认就引入全新架构
- 对整个仓库做大规模风格统一或格式清洗
